# Bugfix 修复流程

本文定义由 Human Admin 提交 Bug、Agent 修复、自动测试、人工验收、独立审查、合并和飞书通知的标准流程。流程兼容 Codex、Grok 及其他能够读取仓库、执行命令和操作 Git/PR 的 Agent。

本流程是 [`multi-agent-coding-v2.md`](./multi-agent-coding-v2.md) 的 Bugfix 专项补充。未明确覆盖的内容继续遵守主流程、`AGENTS.md`、总 TODO 和当前单 TODO。

## 1. 适用范围

适用于以下场景：

- 开发或验收阶段发现的普通 Bug。
- 同一个 Bug 需要多轮 Agent 修复和人工复测。
- 人工验收通过后，由 Agent执行完整门禁、Reviewer 审查和合并到 `dev`。
- 生产紧急 Hotfix，但稳定分支仍须 Human Admin 单独明确授权。

不适用于尚未确认预期行为的功能需求。预期不清晰时先进入需求澄清或 `needs-replan`，不得把猜测直接实现为修复。

## 2. 核心规则

1. 文档先行：修改代码前必须创建或更新单 TODO。
2. 一个修复闭环对应一个 TODO、一个任务分支和一个 PR。
3. 同一个 Bug 的多轮修复继续使用原 TODO、原分支、原 PR 和原 Coder/Fixer 上下文。
4. 同一项目默认只允许一个 active 编码 PR。
5. 普通 Bug 从最新 `dev` 创建 `bugfix/<TODO-ID>-<slug>` 任务分支，PR 目标为 `dev`。
6. 项目已有缺陷分支命名约定时必须沿用；默认使用 `bugfix/<TODO-ID>-<slug>`。
7. 候选修复必须形成提交，人工验收必须绑定准确的候选提交 SHA 和测试环境。
8. 人工验收是 Bugfix 的任务级阻塞门禁。未收到 Human Admin 对指定 SHA 的明确通过结论，不得进入最终合并。
9. 人工验收通过不代替自动测试、独立 Reviewer 和最终门禁。
10. `dev` 可在明确授权、Reviewer `approve` 和门禁通过后自动合并；`main` / `master` 必须由 Human Admin 单独明确授权。
11. 禁止直接删除文件；确需删除时移入 `.delete-pending/` 并记录建议删除命令。
12. 飞书通知只传递状态，不构成验收、审批或合并授权。

## 3. Bug 粒度

多个现象只有同时满足以下条件时，才能放入同一个 TODO 和 PR：

1. 属于同一根因或同一紧密耦合模块。
2. 应作为一个回滚单元。
3. 共享同一套测试与人工验收闭环。

任一条件不满足时，应拆成多个 TODO，并按项目串行 PR 规则依次处理。

多轮复测中出现以下情况时，不再视为原 Bug 的普通修复轮次：

- 暴露独立 Bug。
- 预期行为或验收标准改变。
- 需要修改原 TODO 禁止范围或大幅超出 `allowedPaths`。
- 从局部修复扩大为架构调整、数据库迁移或跨领域修改。
- 达到 `maxRounds` 后仍无法形成稳定方案。

此时进入 `needs-human` 或由 Reviewer 标记 `needs-replan`，不得静默扩大原 PR。

## 4. 启动前只读核查

Agent 收到首次 Bug 提示词后，必须先只读核查：

- 当前分支、HEAD 和工作区状态。
- 当前分支与 `dev`、`main`、远端分支的关系。
- 本地 `dev`、`origin/dev` 和远端 `dev` 是否一致。
- 是否已有同一 Bug 的 TODO、任务分支、Draft PR 或执行记录。
- 总 TODO `currentLock` 和当前 active 编码 PR。
- Bug 是否可复现，实际结果与预期结果是否明确。
- 是否涉及敏感文件、真实删除、生产数据或人工验收。
- 项目规定的测试、构建、文档和工作流脚本是否存在。

核查完成前不得写文件、切换分支、创建分支、提交、推送或创建 PR。

### 4.1 分支判断

| 当前事实                                          | Agent 行为                                               |
| ------------------------------------------------- | -------------------------------------------------------- |
| 已位于同一 Bug 的任务分支                         | 核对 TODO/PR 后继续原流程                                |
| 位于干净的 `dev`                                  | 从最新 `origin/dev` 创建 `bugfix/<TODO-ID>-<slug>`       |
| 位于干净的 `main` / `master`，处理普通 Bug        | 切换并更新 `dev`，再从最新 `origin/dev` 创建 bugfix 分支 |
| 位于 `main` / `master`，且用户明确要求生产 Hotfix | 按第 12 节执行                                           |
| 位于无关分支                                      | 停止自动写入，提供数字选项                               |
| 工作区有来源不明或无关修改                        | 停止自动写入，提供数字选项；不得 stash、覆盖或清理       |
| 已有其他 active 编码 PR                           | 不创建第二个 PR，提示继续、暂停或确认是否属于同一 TODO   |
| detached HEAD、分支关系不明或远端事实不一致       | 切换 `needs-human`，提供核查结果和数字选项               |

### 4.2 数字选项规范

出现歧义时，Agent 提供 2—4 个互斥选项，并说明每个选项的影响。推荐格式：

```text
当前状态：<事实摘要>
阻塞原因：<为什么不能安全继续>

请选择：
1. <推荐操作>（推荐）——<影响>
2. <备选操作>——<影响>
3. 保留当前状态并暂停——不修改文件、分支或 PR

请回复数字；在收到选择前不要继续写入。
```

不得提供 `git reset --hard`、强制推送、直接删除或自动清理未知修改作为默认选项。

## 5. 标准状态流

```text
Bug 输入
  → 只读核查
  → TODO 与任务分支
  → coding
  → 候选提交 + Draft PR
  → needs-human（等待人工验收指定 SHA）
  → fixing（验收失败时循环）
  → needs-human（再次验收新 SHA）
  → reviewing（验收通过）
  → ready-to-merge-dev
  → merged-to-dev
```

状态更新要求：

- 开始修复：任务项标记 `[>]`，锁为 `coding`。
- 候选版本就绪：锁为 `needs-human`，写明候选 SHA、PR、验收步骤和等待对象。
- 人工验收失败：锁恢复为 `fixing`，追加一条 `runs` 记录。
- 人工验收通过：记录验收人、环境、时间和 `acceptedCodeSha`，锁转为 `reviewing`。
- Reviewer 与门禁通过：锁转为 `ready-to-merge-dev`。
- 合并后：任务标记 `[x]`，锁转为 `merged-to-dev`，随后按主流程释放为 `idle`。

只能追加 `runs` 历史，不得覆盖或重写失败轮次。

## 6. 首轮修复与候选版本

Agent 完成首轮修复后：

1. 编写或更新能够复现 Bug 并防止回归的测试。
2. 运行本 Bug 的针对性测试和受影响模块测试。
3. 核对修改范围、敏感文件和测试风险。
4. 提交候选版本并推送任务分支。
5. 创建 Draft PR；已有 PR 时更新原 PR，不创建第二个 PR。
6. 更新单 TODO 执行记录和人工验收要求。
7. 输出候选提交 SHA、分支、Draft PR、测试环境、复现步骤和验收步骤。
8. 将锁切为 `needs-human` 并停止继续修改。

人工验收对象必须可唯一识别：

```text
TODO：<TODO_ID>
分支：<TASK_BRANCH>
候选代码提交：<CANDIDATE_SHA>
Draft PR：<PR_URL>
测试环境：<TEST_ENVIRONMENT>
自动测试：<TARGETED_TEST_RESULTS>
人工验收步骤：<MANUAL_STEPS>
```

用户不能只说“fix 分支测试通过”，必须指明候选 SHA 或确认 Agent 给出的唯一候选 SHA。

## 7. 人工验收失败循环

Human Admin 使用“复测失败”模板提交新证据后，Agent：

1. 确认反馈对应当前 TODO、原任务分支、原 Draft PR 和已测试 SHA。
2. 将锁从 `needs-human` 切换为 `fixing`。
3. 恢复原 Coder 会话作为 Fixer；无法恢复时在原分支核查事实后继续。
4. 只修复反馈对应的阻塞问题，不顺手扩展功能。
5. 运行新的针对性测试和受影响模块测试。
6. 追加执行记录，提交并推送到原分支，更新原 Draft PR。
7. 生成新的候选 SHA，再次切换 `needs-human` 等待验收。

如果反馈属于独立 Bug 或要求扩大规划范围，Agent 停止修改并提供数字选项，不得自行并入原 PR。

## 8. 测试策略

默认采用分层测试，不要求每次小修改都本地重复完整测试。

| 节点                        | 必须执行                                          | 说明                               |
| --------------------------- | ------------------------------------------------- | ---------------------------------- |
| 每轮代码修改后              | 本 Bug 针对性测试                                 | 快速验证根因和回归用例             |
| 候选提交前                  | 针对性测试、受影响模块测试、必要的 lint/typecheck | 确保交给用户的候选版本具备基本质量 |
| Draft PR 推送后             | CI 配置的检查                                     | 可与人工验收并行，但失败必须处理   |
| 人工验收通过后              | 完整项目门禁                                      | 必须针对最终代码事实执行           |
| Reviewer/Fixer 或冲突解决后 | 针对性测试 + 完整门禁                             | PR HEAD 变化后原门禁结果失效       |
| 高风险修复合并后            | 核心冒烟测试                                      | 认证、数据库、全局配置等建议执行   |

本项目完整门禁至少包括：

```bash
npm run lint
npm run typecheck
npm test
npm run build
npm run fm
npm run lm
npm run workflow:check
```

当前 TODO 规定的集成测试、E2E、迁移、备份恢复或其他专项检查也必须执行。

以下修改建议每轮扩大测试范围，必要时每轮执行完整门禁：

- 身份认证、Session、CSRF 和权限。
- 数据库模型、迁移和生产数据处理。
- 全局配置、依赖、构建和部署。
- 缓存、公共组件和跨页面基础能力。
- 上传、外部存储和安全策略。

完整测试较快时可以每轮执行；完整测试成本较高时，本地每轮运行针对性测试，并让 Draft PR CI 异步提供全局反馈。

## 9. 人工验收通过与最终冻结

Human Admin 必须使用“验收通过并授权合并 `dev`”模板，明确：

- 已验收的 `acceptedCodeSha`。
- 测试环境和验收结果。
- 授权 Agent 执行完整门禁、独立 Reviewer 和合并到 `dev`。
- 此授权不包含 `main` / `master`。

收到授权后，Agent 先核对：

1. `acceptedCodeSha` 属于当前任务分支和当前 PR。
2. 自该 SHA 之后，运行时代码、配置、依赖和数据库内容是否变化。
3. 如果只有 TODO、文档或测试结果记录变化，可保留人工验收结论。
4. 如果运行时代码、配置、依赖、迁移或数据处理逻辑变化，人工验收立即失效，必须生成新候选 SHA 并重新验收。

不得用“修改很小”替代重新验收判断。

## 10. 最终 Reviewer、门禁与合并

人工验收有效后：

1. 更新 TODO 和总 TODO 人工验收记录。
2. 在最终 PR HEAD 上执行完整门禁。
3. 创建独立只读 Reviewer；Reviewer 不得由当前 Coder 直接自我批准。
4. Reviewer 给出 `approve`、`request changes`、`comment` 或 `needs-replan`。
5. `request changes` 时恢复原 Fixer；任何运行时代码变化都需要新的人工验收。
6. Reviewer `approve` 后，Supervisor 复核 PR、CI、评论、TODO、高风险和删除状态。
7. 门禁全部通过后 squash 合并到 `dev`。
8. 更新总 TODO、单 TODO、合并提交和项目锁，并同步本地 `dev`。
9. 高风险修复按 TODO 执行合并后冒烟测试。
10. 发送飞书合并结果通知。

GitHub 合并 PR 已更新远端 `dev`，不得再用本地 `dev` 强制覆盖远端。

## 11. 最终门禁失败处理

Agent 先基于证据分类，不把所有失败直接交给用户：

| 分类                                 | 处理方式                                                     |
| ------------------------------------ | ------------------------------------------------------------ |
| 当前修改引入，修复明确且不越界       | Agent 在原分支修复并重新执行测试                             |
| 修复导致运行时代码事实变化           | 生成新候选 SHA，回到人工验收                                 |
| 基线已有失败                         | 记录复现命令、日志摘要及与当前 diff 的关系，交 Reviewer 判断 |
| 需要扩大 allowedPaths 或修改其他领域 | 切换 `needs-human`，提供数字选项                             |
| 根因或验收标准错误                   | Reviewer 标记 `needs-replan`                                 |
| 达到 maxRounds 或同一阻塞重复出现    | 保存检查点，切换 `needs-human`                               |

推荐数字选项：

```text
1. 更新 TODO 范围，在当前分支继续修复
2. 保留当前 PR，进入 needs-replan 并设计替代方案
3. 保留当前状态并暂停
4. 终止本次方案，保留分支、PR 和执行记录供审计
```

在 Human Admin 选择前不得继续扩大修改。

## 12. 生产 Hotfix

生产故障只有在 Human Admin 明确选择 `production-hotfix` 时才走本节：

1. 核对 Bug 是否存在于稳定分支，以及 `dev` 是否已经修复或存在不同实现。
2. 从最新 `main` / `master` 创建独立 Hotfix 分支。
3. 创建 Hotfix TODO，记录生产影响、回滚方案、备份和人工验收。
4. 修复、针对性测试、完整门禁、独立 Reviewer 和生产验收。
5. 等待 Human Admin 使用“稳定分支合并授权”模板明确授权。
6. 授权后才允许合并稳定分支。
7. 稳定分支合并完成后，通过独立 PR 将修复同步回 `dev`；不得假设两个分支自动一致。
8. 执行生产冒烟、监控和飞书结果通知。

普通 Bug 即使 Agent 当前位于 `main`，仍应切换到最新 `dev` 创建任务分支，不能把当前所在分支等同于修复目标。

## 13. 飞书通知

飞书通知使用 `nm-notify-feishu` 的确定性脚本，不直接拼接 Webhook 请求。

首次实际使用或投递失败时运行 `doctor`；使用 `list` 获取目标名称。只有用户要求真实投递测试时才能运行 `test`。

建议通知节点：

| 状态                           | 类型与颜色                 | 是否建议发送 | 内容                                 |
| ------------------------------ | -------------------------- | ------------ | ------------------------------------ |
| 候选版本等待人工验收           | `card` / `blue`            | 可选         | TODO、PR、候选 SHA、验收步骤         |
| `needs-human` / `needs-replan` | `card` / `yellow` 或 `red` | 必须         | 阻塞、影响和所需选择                 |
| 合并到 `dev`                   | `card` / `green`           | 必须         | TODO、PR、门禁、合并提交、高风险摘要 |
| 请求稳定分支授权               | `card` / `purple`          | 必须         | PR、发布风险、明确要求人工授权       |
| 生产 Hotfix 失败               | `card` / `red`             | 必须         | 故障阶段、回滚状态和人工动作         |

通知约束：

- 只在状态转换和人工决策节点通知，不发送分钟级心跳。
- 通知送达、已读或群聊回复均不自动等同于当前 Agent 会话中的授权。
- 不在消息、日志、TODO 或 PR 中输出 Webhook、签名密钥、完整请求体或授权值。
- 配置只能位于 `~/.config/notify-feishu/config.json`，权限必须为 `600`。
- 投递失败只报告目标名称、执行阶段和飞书错误信息。
- 等待人工决策时投递失败仍保持 `needs-human`；合并后通知失败不回滚已完成合并，但必须报告通知失败。

## 14. 人工提示词入口

可直接粘贴的提示词位于 [`../prompts/bugfix/`](../prompts/bugfix/README.md)：

1. 首次提交普通 Bug。
2. 人工复测失败并继续修复。
3. 人工验收通过并授权合并 `dev`。
4. 对门禁失败或范围扩大的人工决策。
5. 中断后恢复。
6. 暂停或终止当前方案。
7. 启动生产 Hotfix。
8. 明确授权合并稳定分支。

Human Admin 只需替换模板中的 `<...>` 占位符。Agent 若不具备某项工具或 skill，必须明确报告限制，不得伪造执行结果。

## 15. 完成条件

Bugfix 只有同时满足以下条件才能标记完成：

- Bug 回归测试存在并通过。
- Human Admin 验收指定代码 SHA 通过。
- 最终运行时代码未超出已验收代码事实，或已重新验收。
- 完整门禁通过。
- 独立 Reviewer `approve`。
- TODO、PR 和高风险记录完整。
- 不存在真实删除和未解决阻塞评论。
- PR 已按规则合并到目标分支。
- 飞书结果通知成功，或已明确报告通知失败。
