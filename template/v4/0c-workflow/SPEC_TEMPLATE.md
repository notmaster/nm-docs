# Spec Template

Spec 是管理员与 Agent 的共同合约：写清事实、边界和验收，不写实现细节噪音。

## 存放与命名

- 目录：`0a-docs/0a-spec/`。
- 命名：`SPEC-<slug>-V<n>.md`，例如 `SPEC-remote-workbench-V1.md`。
- 大版本变更新建 `V<n+1>` 文件并保留旧版本；不要静默改写已确认的合约。
- 草稿可以在 Obsidian 等处撰写，定稿副本必须进入仓库并由 git 管理。

## Frontmatter

```yaml
---
project: my-project
version: V1
status: draft          # draft | confirmed；只有 confirmed 允许执行
execution_mode: staged # staged | auto；启动指令可覆盖
permissions: bypass    # 仅 auto 模式生效：bypass | sandbox，可省略，默认 bypass
verify_entry: ./0d-scripts/verify.sh # 可省略，默认此值
---
```

## 必备章节

按顺序组织正文；标注（可选）的章节按项目复杂度取舍：

1. 背景与总目标：为什么做，做成什么样。
2. 非目标：首版明确不做的事，防止范围蔓延。
3. 已确认决策：所有已拍板的产品、技术与部署决策，逐条列出。
4. 总体架构（可选）：组件图或流程图，复杂项目建议提供。
5. 关键流程与规则：核心用户流程、功能规则、数据边界、安全边界。
6. 实施阶段（可选）：见下节写法；没有本章节时由 Agent 拆分阶段。
7. 最终验收标准：整体完成的判定条件。
8. 待执行前环境确认项：实施时需要检查的环境事实，不是需求疑问。

## 实施阶段的写法

每个阶段必须包含目标与验收；验收拆成机器可验与人工两栏：

```text
### Phase 1: <名称>

目标：

- <本阶段要完成什么>

Verify:

- <机器可验命令，例如测试、构建、curl 冒烟>

Manual:

- <人工验收项；没有则写 none>
```

## 编写规则

- 决策必须收敛：未决问题放"待执行前环境确认项"，不要散落在正文里。
- 每条验收尽量机器可验；纯主观判断项才进 `Manual:`。
- 安全边界必须显式写出：允许操作的路径、密钥处理、禁止暴露的服务面等。
- 阶段划分以"可独立执行、可独立验证"为准，不追求数量均匀。
- `auto` 模式要跑的 Spec，安全边界和停止条件必须完整，因为执行期没有人工审批兜底。
