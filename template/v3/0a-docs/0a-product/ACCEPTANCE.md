# Acceptance Criteria

本文档由管理员维护，用于描述“什么情况下算完成并可以通过验收”。它不替代自动化测试；它负责把产品、设计、后台、部署和安全等人工验收点写清楚。

## 全局通过标准

- 项目可以按 README 或项目说明在本地启动。
- `./0d-scripts/verify.sh` 通过。
- 核心用户路径可以完整走通。
- 页面没有明显布局错位、文字溢出、遮挡、横向滚动或无法点击的关键操作。
- 实现没有明显偏离 `0a-docs/0a-product/REQUIREMENTS.md`。
- 实现符合 `0a-docs/0b-design/DESIGN.md` 的主要视觉规则。

## 管理后台验收

如果项目包含后台，请填写本节；如果没有后台，写“本项目首版不包含管理后台”。

- 管理员可以登录和退出。
- 未登录用户不能访问后台。
- 不同角色只能访问被授权的功能。
- 管理员可以新增、编辑、删除或发布指定内容。
- 表单校验和错误提示清晰。
- 敏感操作有确认或防误操作设计。

## 设计验收

- 色彩、字体、间距、圆角和组件样式符合 `DESIGN.md`。
- 主按钮、输入框、导航、卡片、列表等组件在不同页面保持一致。
- 动效不影响阅读、操作或性能。
- 移动端和桌面端都有清晰的信息层级。
- 不出现未经确认的视觉风格偏移。

## 部署验收

- 项目可以部署到目标平台。
- 生产环境关键页面可访问。
- 必要环境变量和密钥配置有说明。
- 免费额度、计费风险或平台限制已明确。
- 上线后 smoke test 路径清晰。

## 安全验收

- 管理后台和敏感接口有鉴权。
- 客户端不暴露服务端密钥、Webhook secret 或管理 token。
- 用户输入、富文本、Markdown、文件上传或 URL 参数经过安全处理。
- 生产环境关闭调试开关和详细错误输出。
- 上线前完成安全审查，阻断项已修复或由管理员明确接受风险。

## 不通过条件

- 核心路径无法完成。
- 必需自动化验证失败。
- 移动端或桌面端存在严重布局问题。
- 关键功能未覆盖 `REQUIREMENTS.md` 的首版范围。
- 实现明显偏离 `DESIGN.md`。
- 管理后台或敏感数据存在未授权访问风险。
- 部署目标无法运行首版功能。
